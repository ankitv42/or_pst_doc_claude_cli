# Cost vs. Quality Tradeoffs in AI Features

## What Is It? (Plain English)

Every AI feature sits on a spectrum between cheap-and-acceptable and expensive-and-excellent. The decision of where to sit on that spectrum is one of the most important product and engineering decisions you will make, and it is rarely as simple as "use the best model available." The cost of an LLM call is not just the API fee — it is also latency (users waiting longer), infrastructure complexity, rate limits, and the operational burden of managing multiple model integrations.

The core insight is that different parts of your AI pipeline have different quality requirements. In ORCA, Agent 3's capital scoring formula needs to produce a numerically correct score that directly affects whether an order is auto-approved or escalated. That is a high-stakes decision. Agent 1's demand summary, which is one input among several, might tolerate a 15% error rate without meaningfully changing the final recommendation. If you treat every step the same and use the most expensive model everywhere, you waste money. If you use the cheapest model everywhere, you introduce quality failures at the critical steps where they matter most.

**Caching** is the most underused cost-reduction strategy in LLM applications. If the same policy document is being retrieved and injected into the Agent 2 prompt on every pipeline run, and that document is 2,000 tokens long, you are paying for 2,000 tokens of input processing on every single call. With prompt caching (available in Anthropic's API and configurable in OpenAI), the model keeps the static portion of the prompt in a cached state, and you only pay a fraction of the price for repeated calls with the same prefix. For ORCA, which uses Groq's free tier, caching is less of a financial issue today — but at production scale on paid APIs, prompt caching can reduce costs by 60–80% on RAG-heavy pipelines.

## Why Google Cares About This

Google operates products at a scale where every penny of cost per query multiplies across billions of requests. The discipline of cost-quality tradeoff is not just a startup concern — at Google, a 10% reduction in inference cost for a major AI feature can save tens of millions of dollars annually. Interviewers expect senior candidates to have concrete intuition about model costs, to have made deliberate tradeoff decisions on past projects, and to articulate a framework for when to upgrade versus when to optimize. Saying "we just used GPT-4 for everything because it was easiest" is a red flag. Saying "we profiled the pipeline by step, identified Agent 3 as the critical path for quality, and allocated budget accordingly" is what they want to hear.

## Interview Questions & Answers

### Q1: How would you decide between using GPT-4o vs. a smaller model like GPT-4o-mini for an AI pipeline?

**Answer:** I would never make this decision at the pipeline level — I would make it step-by-step within the pipeline, because different steps have different quality requirements. The first thing I do is map each step on two axes: how much does the quality of this step affect the final output, and how often does this step run. Steps that are high-impact and high-frequency are the ones where model quality most directly affects the product's value proposition. Steps that are low-impact or low-frequency can use cheaper models.

For ORCA, the capital scoring step (Agent 3) is the one that most directly determines whether an order is auto-approved or escalated. A scoring error here has a direct financial consequence — either an order that should have been reviewed gets auto-approved, or a routine order unnecessarily consumes a human reviewer's time. I would use GPT-4o or Claude Sonnet for Agent 3, because the cost differential between a $0.002 call and a $0.015 call is trivial compared to the consequence of a bad routing decision.

For Agent 1's demand analysis, I would use a smaller, faster model. Agent 1's output is one input among four into the pipeline, and the pipeline has multiple downstream quality checks. A moderately imprecise demand trend analysis that still captures the direction of demand change (up/flat/down) is sufficient for the pipeline to produce a good overall recommendation. Using GPT-4o for Agent 1 would triple the API cost without materially improving the auto-approval acceptance rate.

The way I validate this reasoning after the fact is by running ablation experiments: run the pipeline with the expensive model on all steps, then replace Agent 1 with the cheap model and measure whether the overall auto-approval acceptance rate changes. If the change is within noise, you have found a cost reduction with no quality impact. If it degrades meaningfully, you have discovered that Agent 1 quality matters more than you thought, and you keep the expensive model.

### Q2: When does caching LLM responses make sense, and how do you implement it?

**Answer:** Caching LLM responses makes sense when the same input (or the same input prefix in prompt caching) produces the same output, and when that input appears frequently enough that the savings outweigh the engineering cost of building and maintaining the cache. There are three categories of caching worth distinguishing: semantic caching, exact caching, and prompt-prefix caching.

Exact caching is the simplest: if you hash the full prompt and it matches a previous call, return the cached response. This works for structured analytical queries — for example, in ORCA, if two pipeline runs process the same SKU with the same inventory snapshot, Agent 2 would produce the same options. You can cache that response for a configurable TTL (time-to-live). The engineering implementation is straightforward: a dictionary (in memory) or Redis (distributed) keyed on a hash of the prompt.

Semantic caching is more sophisticated: if a new query is semantically similar to a previous query, return the cached response. This requires storing query embeddings and doing a nearest-neighbor lookup before calling the LLM. It is useful for RAG retrieval caching — if two pipeline runs ask essentially the same retrieval question, you avoid a second embedding lookup and database query. The tradeoff is that semantic similarity has a threshold: too strict and you rarely get cache hits; too loose and you return an answer to a slightly different question.

Prompt-prefix caching (available natively in Anthropic's API via cache_control breakpoints, and in OpenAI) is the highest-leverage option for RAG-heavy pipelines. The pattern is: put the static content (system prompt + retrieved documents) early in the prompt, mark it as cacheable, and append the dynamic content (current SKU data) at the end. The first call is full price. Subsequent calls with the same static prefix are 60-90% cheaper. For ORCA, the policy documents retrieved for Agent 2 are largely the same across runs; prompt-prefix caching would be the correct approach at production scale.

### Q3: How do you calculate the ROI of an AI feature when the quality improvement is hard to measure?

**Answer:** The starting point is to distinguish between the costs that are easy to quantify (API fees, engineering time, infrastructure) and the benefits that require more creative measurement (decision quality improvement, time saved, errors prevented). For the costs, a straightforward tally usually works. The difficulty is always on the benefit side.

For ORCA, the measurable benefit is hours saved in inventory analyst time: if the system auto-approves 70% of decisions that previously required a full human review cycle, and each review takes 20 minutes, you can calculate the analyst time saved per month and multiply by the loaded cost of that analyst. This gives you a floor estimate of benefit.

The harder benefit to quantify is error reduction. If the AI improves the quality of reorder decisions — ordering the right quantity at the right time — the financial impact is the reduction in holding cost (from over-ordering) plus the reduction in stockout cost (from under-ordering). These can be estimated from historical order data: what was the average holding cost per SKU-month and what was the average stockout cost per incident before the AI system.

When the quality improvement cannot be measured at all before launch, I use a pilot approach: run the system in shadow mode (AI makes recommendations, humans make final decisions as they always did), collect AI recommendations versus human decisions over 4-6 weeks, and then evaluate the AI recommendations against the actual outcomes. This gives you an offline quality estimate before you commit to a full build. If the AI recommendations, evaluated against outcomes, look at least as good as historical human decisions, you have a reasonable case for positive ROI — even before measuring it in production.

### Q4: What is the build-vs-buy-vs-fine-tune decision for LLM models in an enterprise AI system?

**Answer:** The decision framework I use has three sequential questions. First: does a general-purpose frontier model (GPT-4o, Claude Sonnet, Llama 3.3) handle the task well enough out of the box? If yes, use it — the engineering and maintenance cost of fine-tuning or building your own model is almost never justified when a general model works. Second: if the general model is not good enough, is the gap due to lack of domain knowledge, or due to the model not following the required output format? If it is domain knowledge, fine-tuning may help. If it is format, better prompting or structured output constraints (JSON mode, tool calling) usually fixes it. Third: is the data volume and quality required for fine-tuning actually available, and does the team have the capacity to maintain a fine-tuned model over time?

For ORCA, the right answer at this stage is clearly "use an existing API." The task — analyzing inventory data and applying policy constraints — is well within the capability of llama-3.1-8b-instant with good prompting. The policy retrieval via RAG is more important to quality than model size. There is no inventory-domain training data large enough to justify fine-tuning, and the operational burden of maintaining a fine-tuned model on a free-tier stack would be disproportionate to the benefit.

The scenario where fine-tuning becomes compelling is when you have thousands of labeled examples of the specific task, when the task is highly formulaic (e.g., extracting structured data from a specific document format), and when latency or cost requirements make frontier model APIs prohibitive. In enterprise inventory management at a large retailer with 50,000+ SKUs, fine-tuning a smaller model on historical approved/rejected decisions might eventually make sense — but only after you have a working system that produces the training data.

### Q5: How do you decide when an AI feature has crossed from "worth the cost" to "too expensive to run"?

**Answer:** The decision threshold is when the per-decision cost of running the AI exceeds the value the AI adds to that decision. This sounds obvious but requires operationalizing two things that organizations often leave undefined: what is the value of an AI-assisted decision, and what is the unit of cost measurement.

For ORCA, the value of an AI-assisted inventory decision is primarily the time saved by a human reviewer. If a human analyst would otherwise spend 20 minutes reviewing a reorder recommendation and the AI provides a recommendation that takes the analyst 3 minutes to validate, the AI saved 17 minutes. At a loaded cost of $45/hour, that is $12.75 of value per decision. If the pipeline costs $0.30 in API fees, the ROI is 42:1 and the feature is clearly worth running. If the API fees were $15 per run, the ROI would be negative and you would need to either reduce the model cost or challenge whether the AI-assisted review is actually faster than an unassisted one.

The harder case is when you cannot measure the time saving directly. In that situation, I use a conservative proxy: count the number of human approval decisions per month, estimate that the AI saves at least one third of the review time per decision (a conservative estimate), and compare the resulting dollar figure to the API cost. If the API cost is less than half of the conservative estimated saving, the feature is worth running. If the API cost exceeds the conservative estimate, you either need to reduce cost or run a time-tracking study to get a more precise saving estimate.

The practical trigger for re-evaluating is when the API cost per run increases significantly (due to model price changes or prompt length growth) or when the auto-approval rate drops significantly (meaning the AI is generating more escalations that require human review, reducing its automation value). Either event should trigger a cost-quality review.

## Key Points to Say in the Interview
- Make model selection decisions at the step level, not the pipeline level — different steps have different quality requirements
- Caching (exact, semantic, and prompt-prefix) is the highest-leverage cost reduction in RAG-heavy pipelines
- ROI = (hours saved × rate + errors reduced × cost per error) minus (API cost + engineering overhead)
- Fine-tuning only makes sense when a general model fails on the task AND you have labeled data AND the team can maintain it
- The threshold question is: does per-decision AI cost exceed per-decision AI value?
- Validate cost assumptions with ablation experiments before committing to a model choice

## Common Mistakes to Avoid
- Using the same model tier for all pipeline steps without analyzing which steps are quality-critical
- Ignoring prompt length as a cost driver — long RAG contexts significantly increase token costs
- Confusing "we used a better model" with "we improved the product" — better model + poor prompting often underperforms a good prompt on a smaller model
- Skipping ROI analysis and building features that cost more to run than the value they deliver
- Treating fine-tuning as a default improvement path — it requires significant data and maintenance investment
